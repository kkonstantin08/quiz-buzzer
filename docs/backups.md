# Резервные копии production-данных

Скрипты сохраняют production SQLite-базу из `backend_data` и пользовательские загрузки из `backend_uploads`. Для базы используется команда SQLite `.backup`, а не копирование работающего файла.

## Настройка

Задайте эти host-side значения в `.env` рядом с `docker-compose.yml`:

```env
BACKUP_DIR=./backups
BACKUP_RETENTION_COUNT=14
```

`BACKUP_DIR` — каталог на хосте, смонтированный в maintenance-контейнеры как `/backups`. По умолчанию архивы лежат в `<project>/backups/`, вне двух Docker volume с данными. Каталог должен быть доступен на запись пользователю, который запускает Docker Compose. `BACKUP_RETENTION_COUNT` сохраняет 14 последних обычных архивов; ротация выполняется только после успешной проверки нового архива. Аварийные архивы перед восстановлением сохраняются и автоматически не удаляются.

## Создание и просмотр копий

Выполняйте команды из корня репозитория на production-сервере:

```bash
./scripts/backup.sh
ls -lh "${BACKUP_DIR:-./backups}"
```

Обычный архив называется `quiz-buzzer-backup-<UTC-дата-время>-<pid>.tar.gz`. Соседний файл `.sha256` обязателен для восстановления. Внутри находятся `database.sqlite`, `uploads/` и `metadata.json` с UTC-временем создания и коротким Git SHA, если он доступен в checkout.

Скрипт проверяет, что архив не пустой и распаковывается, сверяет SHA-256 и требует от `PRAGMA integrity_check` ровно `ok`. При любой ошибке он завершится с ненулевым кодом и не затронет существующие копии.

## Восстановление

Восстановление останавливает backend и заменяет production-базу и uploads. Сначала выберите архив внутри `BACKUP_DIR`:

```bash
./scripts/restore.sh ./backups/quiz-buzzer-backup-YYYYMMDDTHHMMSSZ-PID.tar.gz
```

Введите `RESTORE` в приглашении. Для контролируемого автоматического запуска используйте явный флаг:

```bash
./scripts/restore.sh ./backups/quiz-buzzer-backup-YYYYMMDDTHHMMSSZ-PID.tar.gz --yes
```

До остановки backend скрипт проверяет архив, checksum, распаковку и целостность SQLite. После остановки он создаёт `emergency`-архив текущих данных, восстанавливает выбранную копию, запускает backend и запрашивает `/api/health` внутри backend-контейнера. Если поздний шаг завершится ошибкой, аварийный архив останется на месте, а скрипт выведет инструкцию по восстановлению; не удаляйте его, пока не проверите сервис.

## Ежедневный cron

Настройте cron для того же пользователя хоста, который может выполнять `docker compose`, и используйте абсолютные пути:

```cron
0 3 * * * cd /srv/quiz-buzzer && /srv/quiz-buzzer/scripts/backup.sh >> /var/log/quiz-buzzer-backup.log 2>&1
```

На целевом диске должно хватать места и для базы, и для uploads. Регулярно проверяйте лог и список архивов.

## Внешнее хранение

Архивы на том же сервере **не** защищают от полной потери сервера, отказа диска или компрометации хоста. Это только локальный слой восстановления. Следующей операционной задачей должна стать передача проверенных архивов во внешнее зашифрованное хранилище и тестовое восстановление из него.
